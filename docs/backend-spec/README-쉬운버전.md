## 0. 공통 사항

> 💡 **공통 사항** = 앱 전체에 걸쳐 동일하게 적용되는 약속. 모든 화면이 따라야 하는 규칙.

### 0.1 사용자 역할 (Role)

> 💡 **Role(역할)** = 한 사용자가 앱 안에서 어떤 입장인지 구분하는 라벨. "당신은 손님이세요? 일하는 사람이세요?" 같은 거.

- `customer` — 서비스 요청자 (가사/운전/심부름 의뢰)
- `helper` — 가사도우미 (ART)
- `driver` — 운전자 (고객 차량을 모는 대리/일일 기사)
- `errand` — 심부름꾼 (예정/검토)

> ⚠️ **합의 필요**: 한 사용자가 여러 역할 동시에 가질 수 있게 할지 (예: 평일엔 헬퍼, 주말엔 드라이버). 지금 프론트는 가입할 때 하나만 고르도록 만들어져 있음.

---

### 0.2 다국어

> 💡 **다국어 (i18n)** = 같은 화면을 여러 언어로 보여주는 기능. internationalization을 줄여서 i18n이라 부름 (i + 18글자 + n).

- 지원 언어: `id`(인도네시아·기본), `en`, `ko`, `zh`, `ja`
  - 💡 `id` = 인도네시아어, `en` = 영어, `ko` = 한국어, `zh` = 중국어, `ja` = 일본어. **2글자 국가 코드** 약속.
- 모든 사용자 입력(이름, 자기소개, 리뷰 등)은 **원문 그대로 저장**.
  - 💡 "원문 그대로 저장" = 사용자가 인니어로 쓴 자기소개를 한국어로 자동 번역하지 않고, 인니어 그대로 DB에 넣는다는 뜻.
- 시스템 메시지·카테고리 마스터만 **번역 키 보유**.
  - 💡 **번역 키** = "안녕하세요"를 직접 저장하지 않고 `hello_message` 같은 식별자로 저장. 언어 바뀌면 그 식별자로 알맞은 번역문을 찾아 보여줌.
  - 💡 **마스터 데이터** = 시스템이 기준으로 쓰는 고정 데이터 (예: 활동 카테고리 "청소·요리·세탁" 같은 목록). 사용자가 만드는 콘텐츠와 구분.

---

### 0.3 통화·지역

- 통화: **IDR (Rp)** — 정수 단위 저장 권장
  - 💡 **IDR** = 인도네시아 루피아 (인니 화폐 단위 코드). USD가 미국 달러인 것과 같은 식.
  - 💡 **정수 단위 저장** = 25,000.50 같은 소수점 없이 25000으로만 저장하라는 뜻. 루피아는 소수점이 거의 의미 없어서.
- 기본 지역: **자카르타 (Jakarta)** — 기본 지도 중심: Kemang/Kebayoran Baru
- 좌표계: **WGS84 (lat/lng)**
  - 💡 **WGS84** = 전 세계 표준 위치 좌표계. 구글 지도·아이폰 GPS 다 이걸 씀. 그냥 "보통 쓰는 위도/경도"로 이해하시면 됨.
  - 💡 **lat/lng** = 위도(latitude) / 경도(longitude). 지구 위 한 지점을 두 숫자로 표시.

---

### 0.4 인증

> 💡 **인증 (Authentication)** = "이 사람이 누구인지" 확인하는 절차. 로그인 같은 거.

- 회원가입/로그인 (이메일·전화번호 기준 — 정책 결정 필요)
  - ⚠️ **합의 필요**: 인니에선 이메일 안 쓰는 사람 많음. 전화번호 인증이 더 일반적.
- **토큰 기반 세션 (JWT 예상)**
  - 💡 **토큰** = 로그인하면 서버가 발급해주는 "출입증". 이후 모든 요청에 이 출입증을 첨부해서 보냄.
  - 💡 **JWT** ("제이덜뜨") = 가장 흔한 토큰 형식. 글자 묶음 하나로 사용자 정보 + 만료 시간 다 들어있음.
  - 💡 **세션** = 로그인부터 로그아웃까지의 한 번의 사용 단위. "지금 로그인된 상태"를 의미.
- 약관 동의 이력 보관 필수
  - 💡 사용자가 언제 어떤 약관에 동의했는지 DB에 기록 — 나중에 문제 생겼을 때 증거로 씀.

---

### 0.5 미디어

> 💡 **미디어** = 사진·동영상·파일 같은 큰 데이터. 텍스트와 구분.

- 프로필 사진, 게시물 이미지, 채팅 첨부 이미지
- **CDN/스토리지 업로드 흐름 (Pre-signed URL 패턴 권장)**
  - 💡 **CDN** (Content Delivery Network) = 이미지·파일을 전 세계 여러 서버에 미리 뿌려놔서 사용자랑 가까운 곳에서 빠르게 받게 해주는 시스템. 유튜브 영상 빠른 이유도 CDN.
  - 💡 **스토리지** = 파일을 보관하는 창고 (예: AWS S3).
  - 💡 **Pre-signed URL** = "미리 서명된 URL". 백엔드가 "이 주소로 30분 안에 사진 올려도 돼" 라는 임시 권한 링크를 발급해서, 앱이 그 URL에 직접 사진을 업로드. **백엔드를 거치지 않고 사진을 S3에 바로 보내서** 서버 부담을 줄이는 방식.

## 2. 데이터 모델 개요 (초안)

> 💡 **데이터 모델** = DB(데이터베이스)에 어떤 종류의 데이터를 어떤 모양으로 저장할지 정한 설계도.
> 💡 **엔티티 (Entity)** = 데이터의 한 종류. "User, Helper, Booking" 각각이 하나의 엔티티. DB의 표 한 개라고 생각해도 됨.
> 💡 **필드 (Field)** = 엔티티 안의 항목. User 엔티티 안에 id·name·email 같은 필드들이 있음. 표의 열(컬럼)에 해당.

| 엔티티 | 핵심 필드 | 비고 |
|---|---|---|
| `User` | id, role, name, phone, email, lang, createdAt | 역할별 확장 프로필 분리 |
| `HelperProfile` | userId, photo, location, rating, pricePerHour, skills[], experienceYears, isVerified, serviceFrequency | 헬퍼 전용 |
| `DriverProfile` | userId, photo, rating, pricePerHour, vehicleTypes[], driverServices[], licenseClass | 드라이버 전용 |
| `Booking` | id, customerId, partnerId, type, scheduledAt, status, address, totalPrice | 예약·주문 |
| `Review` | id, bookingId, rating, comment, createdAt | 후기 |
| `ChatRoom` / `Message` | … | 채팅 |
| `ErrandRequest` | id, customerId, type, description, location, budget, status | 심부름 게시 |
| `Notification` | id, userId, type, payload, readAt | 알림 |

**자주 보이는 필드 이름 풀이:**
- `id` = 고유 식별 번호. "이 유저 = 12345" 같은 거.
- `userId`, `customerId`, `partnerId` = "어느 User의 ~인가"를 가리키는 연결고리.
- `createdAt` = 만들어진 시각.
- `updatedAt` = 마지막으로 수정된 시각.
- `readAt` = 사용자가 읽은 시각 (안 읽었으면 null).
- `[]` (대괄호) = 배열. 여러 개 들어있다는 뜻. `skills[]`는 "스킬이 여러 개" 라는 의미.
- `rating` = 평점.
- `status` = 상태 (예: "pending/confirmed/completed/cancelled" — 대기·확정·완료·취소).
- `payload` = 그 알림에 담긴 실제 내용 데이터.

**역할별 확장 프로필 분리**의 의미:
- 💡 모든 User는 `User` 표에 기본 정보가 들어가고, 헬퍼면 추가로 `HelperProfile` 표에 헬퍼 전용 정보(시급·스킬 등)가 들어감. 한 사람 = 두 표에 나뉘어 저장. **공통 정보 따로, 역할별 정보 따로** 관리하면 깔끔.

## 3. 각 화면 문서 작성 규칙

각 화면 문서는 다음 섹션 포함:

1. **목적** — 한 줄 요약
2. **진입 경로** — 어디서 들어오는지
3. **스크린샷** — `./screenshots/{화면명}/` 폴더에 이미지 첨부
4. **UI 구성** — 영역별 설명
5. **데이터 / 표시 필드** — 무엇을 보여주는가
6. **사용자 액션** — 버튼·입력·제스처
7. **API 요구사항** — 엔드포인트 후보 (메서드·경로·요청·응답)
   - 💡 **API** = 앱(프론트)과 서버(백엔드)가 대화하는 약속. "헬퍼 목록 줘", "이 사람 정보 줘" 같은 요청·응답 규칙.
   - 💡 **엔드포인트** = API 한 개의 주소. 예: `/api/v1/partners`.
   - 💡 **메서드** = 어떤 종류 요청인지 표시. GET("달라"), POST("만들어달라"), PUT("수정해달라"), DELETE("지워달라").
8. **비즈니스 로직 / 정책 이슈** — 백엔드와 합의 필요한 부분
9. **엣지 케이스** — 빈 상태, 에러, 권한 없음 등
   - 💡 **엣지 케이스** = 흔하진 않지만 발생할 수 있는 예외 상황. "사용자가 인터넷 끊긴 상태에서 버튼 누르면?", "검색 결과가 0건이면?" 같은 거.

## 🤝 백엔드와 회의할 때 이 문서 활용법

1. **회의 전**: 화면별 문서의 "API 요구사항" 섹션 미리 보고 "이런 요청이 가능한가" 백엔드에 던져두기
2. **회의 중**: "비즈니스 로직 / 정책 이슈" 섹션 같이 보면서 결정
3. **회의 후**: 합의된 사항을 문서에 반영 (✅ 또는 ⚠️ 해결됨 표시)

비개발자 입장에선 **"백엔드가 만들어줘야 하는 일감 리스트"** 로 보시면 돼요. 각 ⚠️ 표시는 회의 안건.
