## 1. 목적

고객이 지도 위에서 주변 헬퍼·드라이버를 시각적으로 탐색하고, 필터링·정렬해서 원하는 파트너를 찾아 상세 화면으로 진입한다.

> 💡 **필터링** = 조건에 맞는 것만 골라내기 (예: "정기 헬퍼만 보여줘").
> 💡 **정렬** = 순서대로 늘어놓기 (예: "평점 높은순으로").
> 💡 **파트너** = 헬퍼 + 드라이버를 묶어 부르는 단어. 고객 입장에서 "일해줄 사람".

## 2. 진입 경로

> 💡 **진입 경로** = 이 화면이 어디서 어떻게 열리는지.

- 하단 탭바의 "지도(Map)" 탭
- 홈 화면에서 "지도에서 보기" 또는 서비스 카드 탭 → `serviceType` 파라미터와 함께 진입 가능

**라우트 파라미터:**
> 💡 **파라미터** = 화면을 열 때 같이 넘기는 추가 정보. "이 화면 열되, 이러이러한 옵션으로 열어줘".

- `expanded?: boolean` — 진입 시 바텀시트를 펼친 상태로 표시
  - 💡 **boolean** = true/false (참/거짓) 두 가지 값만 가지는 타입.
  - 💡 **?** = 옵션. 없어도 됨.
  - 💡 **바텀시트** = 화면 하단에서 위로 올라오는 패널 (지도 앱에서 식당 정보가 아래에서 뜨는 그것).
- `serviceType?: 'regular' | 'onetime' | 'live-in'` — 진입 시 서비스 유형 필터 미리 적용
  - 💡 **`'regular' | 'onetime' | 'live-in'`** = "이 세 가지 값 중 하나만 가능"이라는 타입. **유니온 타입**이라고 부름. regular(정기), onetime(단기/일회성), live-in(상주/입주).

## 4. UI 구성

> 💡 **UI** (User Interface) = 사용자가 직접 보고 만지는 화면. **UI 구성** = 화면을 영역별로 어떻게 쪼개서 만들었는지.

### 4.1 상단 검색 바 (오버레이)
> 💡 **오버레이** = 다른 컨텐츠 위에 떠 있는 레이어. 검색바가 지도 위에 떠 있는 형태.

- 검색 입력창: 파트너 이름 또는 위치 키워드
- 클리어(X) 버튼

### 4.2 지도 (Google Maps)
- 커스텀 스타일 적용 (밝은 회색 톤, POI/대중교통 라벨 숨김)
  - 💡 **POI** (Point of Interest) = 지도 위 상점·식당 같은 관심 지점 표시. 우리 지도엔 안 보이게 가렸음.
- 사용자 현재 위치 표시 (권한 허용 시)
- 파트너 마커: 사진 + 이름 + 평점 표시. 헬퍼/드라이버 색상 구분
  - 헬퍼: 앰버 #F59E0B
  - 드라이버: 인디고 #6366F1
  - 💡 **#F59E0B**, **#6366F1** = 색상 코드 (Hex). 컴퓨터가 색을 구분하는 표준 표기.
- 줌 인/아웃 버튼, 내 위치로 이동 버튼

### 4.3 바텀시트 (2단계)

**Peek 상태 (240pt)** — 살짝만 올라온 상태
- 💡 **pt** = 점(points). 화면 크기 단위. 240pt ≈ 화면 1/3 정도.
- 헤더: "n명의 파트너" 카운트 배지
- 필터 칩 행 (가로 스크롤): 헬퍼·드라이버·서비스유형·즉시가능·검증완료·경력5년+·활동·조건·정렬
  - 💡 **칩** = 작고 둥근 버튼 (앱에서 자주 보는 그것). 토글 가능한 작은 라벨.
- 스토리 리스트: 파트너 사진 가로 스크롤 (얼굴+이름+가격)

**Expanded 상태 (화면의 72%)** — 크게 펼쳐진 상태
- 헤더: 카운트 + 정렬 드롭다운
- 파트너 카드 리스트 (FlatList)
  - 💡 **FlatList** = React Native가 제공하는 긴 리스트를 부드럽게 보여주는 컴포넌트. 스크롤 가능한 리스트 그 자체.
- 카드 탭 시 헬퍼/드라이버 상세 화면으로 이동

### 4.4 파트너 선택 미리보기 시트
- 마커 탭 시 하단에서 슬라이드 업
- 사진 / 이름 / 평점 / 시급 / 위치 / 스킬 / 경력 / 검증 배지
- "상세 보기" CTA → WorkerDetailScreen / DriverDetailScreen
  - 💡 **CTA** (Call To Action) = 사용자가 다음 행동을 하도록 유도하는 버튼. "상세 보기", "예약하기" 같은 거.

### 4.5 필터 모달들

1. **활동 카테고리** (다중 선택)
   - cleaning, cooking, laundry, other 카테고리 아래 세부 항목
   - 각 항목은 백엔드 `skill` 필드와 매핑됨
   - 💡 **매핑** = 한 데이터를 다른 데이터에 연결하기. "바닥 청소" 항목 → "Beberes" 스킬에 매핑되어 있어서, 사용자가 바닥 청소 체크하면 백엔드엔 Beberes로 전달.

2. **서비스 유형** (단일 선택)
   - regular(정기) / onetime(단기) / live-in(상주)

3. **헬퍼 조건** (검증 토글 + 경력 라디오)
   - 💡 **토글** = 켜고 끄기 두 가지 상태.
   - 💡 **라디오** = 여러 옵션 중 하나만 선택 (라디오 방송 채널처럼).
   - 검증된 파트너만
   - 경력: 전체 / 1년+ / 3년+ / 5년+

4. **정렬**
   - 평점순 / 후기 많은순 / 가격 낮은순 / 가격 높은순

## 5. 데이터 / 표시 필드 (Partner)

> 💡 **데이터 필드** = 한 파트너에 대해 갖고 있어야 하는 정보 항목들.
> 💡 **타입** = 이 필드가 무슨 종류 데이터인지 (숫자/글자/참거짓 등).

| 필드 | 타입 | 설명 |
|---|---|---|
| `id` | string | 파트너 고유 ID |
| `name` / `firstName` | string | 표시명 |
| `lat`, `lng` | number | 위치 좌표 |
| `rating` | number (0~5) | 평균 평점 |
| `pricePerHour` | number (IDR) | 시급 |
| `isAvailable` | boolean | 현재 즉시 가능 여부 |
| `photo` | string(url) | 프로필 사진 URL |
| `location` | string | 표시용 동네명 (예: Kebayoran Baru) |
| `totalJobs` | number | 누적 작업 건수 |
| `serviceFrequency` | `'regular' \| 'special' \| 'both'` | 정기/단기/둘다 |
| `skills` | string[] | 가능 스킬 목록 |
| `experienceYears` | number | 경력 연수 |
| `isVerified` | boolean | 신원·서류 검증 완료 |
| `partnerType` | `'helper' \| 'driver'` | 파트너 종류 |
| `driverServices?` | string[] | 드라이버 한정 |
| `licenseClass?` | string | 드라이버 한정 (예: SIM A) |

**타입 풀이:**
- 💡 **string** = 글자열 (예: "Sari Dewi")
- 💡 **number** = 숫자 (정수·소수 다 포함)
- 💡 **boolean** = true/false
- 💡 **string[]** = 글자열 여러 개 묶음 (배열). `skills` = ["청소", "요리"] 같은 모양.
- 💡 **`A | B`** = A 또는 B 둘 중 하나만 가능.
- 💡 **?** = 옵션. 없어도 됨.

> ⚠️ **백엔드 합의 필요**: 위 필드는 결국 `HelperProfile` / `DriverProfile` + 집계 통계(`rating`, `totalJobs`, `isAvailable`)의 조합. `isAvailable`은 파트너 본인 토글인지, 시스템이 스케줄 보고 자동 판단인지 결정 필요.
> 💡 **집계 통계** = 여러 건의 데이터를 모아 계산한 값. 평점은 모든 리뷰 평균, totalJobs는 완료 건수 합계 등. DB에서 즉시 계산 vs 미리 계산해서 저장 결정 필요.

## 6. 사용자 액션

> 💡 **사용자 액션** = 사용자가 화면에서 할 수 있는 행동 목록. 각 행동이 어떤 결과를 만드는지.
> 💡 **트리거** = "어떤 행동이 일어났을 때". 버튼 누름·입력·스와이프 등.

| 액션 | 트리거 | 결과 |
|---|---|---|
| 검색 | 검색창 입력 | 파트너 이름·위치 부분 일치 필터 (현재 클라이언트 필터링) |
| 마커 탭 | 지도 마커 클릭 | 파트너 미리보기 시트 표시 |
| 미리보기 → 상세 | "상세 보기" 버튼 | WorkerDetail / DriverDetail로 이동 |
| 파트너 타입 토글 | 헬퍼/드라이버 칩 | 마커·리스트 필터 |
| 서비스 유형 변경 | 조건 모달 | regular/onetime/live-in 필터 |
| 즉시 가능 토글 | 칩 | `isAvailable: true` 만 노출 |
| 검증 토글 | 칩 | `isVerified: true` 만 노출 |
| 경력 5년+ 토글 | 칩 | 빠른 경력 필터 |
| 활동 선택 | 활동 모달 (다중) | 선택된 활동 → skill 매핑 → 필터 |
| 정렬 변경 | 정렬 모달 | 평점/후기/가격 정렬 |
| 내 위치로 | 우측 하단 버튼 | 현재 GPS 좌표로 이동 (권한 필요) |

> 💡 **클라이언트 필터링** = 앱 안에서 데이터 가져온 후 거기서 거르는 방식. 반대는 서버에 "필터된 결과만 줘" 부탁해서 받는 **서버 필터링**. 데이터 적을 땐 클라이언트, 많을 땐 서버가 좋음.

## 7. API 요구사항

> 💡 **API 요구사항** = "백엔드가 만들어줘야 하는 통신 인터페이스 목록". 각각이 백엔드 개발자의 작업 한 건.

### 7.1 파트너 검색 (메인) 🤝

```
GET /api/v1/partners
```

> 💡 **GET** = "달라" 요청. 데이터를 받아오는 용도.
> 💡 **/api/v1/partners** = 이 API의 주소. `/v1`은 버전 1을 의미 — 나중에 큰 변경 있으면 v2로 새 버전 만듦.

**Query 파라미터:**
> 💡 **Query 파라미터** = URL 뒤에 `?` 붙여서 보내는 옵션들. 예: `/partners?partnerType=helper&availableOnly=true`. 검색 옵션 같은 것.

| 이름 | 타입 | 설명 |
|---|---|---|
| `bbox` | string `swLat,swLng,neLat,neLng` | 지도 가시 영역 |
| `lat`, `lng`, `radius` | number | 또는 반경 검색 |
| `partnerType` | `helper \| driver \| all` | 기본 `all` |
| `serviceType` | `regular \| onetime \| live-in` | 선택 |
| `availableOnly` | boolean | |
| `verifiedOnly` | boolean | |
| `minExperience` | number (years) | 0/1/3/5 |
| `skills` | string[] (반복) | 활동 → skill 매핑 결과 |
| `keyword` | string | 이름·위치 검색 |
| `sortBy` | `rating \| reviews \| price_low \| price_high` | 기본 `rating` |
| `cursor` / `limit` | 페이지네이션 | 리스트 뷰용 |

**용어 풀이:**
- 💡 **bbox** (bounding box) = 지도에서 화면에 보이는 사각형 영역. 4개 좌표 (남서 + 북동)으로 표시. "지금 화면에 보이는 곳만 검색해줘"라는 의미.
  - sw = south-west(남서), ne = north-east(북동)
- 💡 **radius** = 반경. 한 점에서 N미터 안의 결과.
- 💡 **페이지네이션 (Pagination)** = 검색 결과가 많을 때 한 번에 다 안 보내고 페이지 단위로 쪼개서 보내기. "1페이지 20개, 2페이지 20개..." 식.
- 💡 **cursor 방식 vs offset 방식**:
  - **offset** = "20번부터 20개" (페이지 번호 방식). 사용자가 스크롤하다 중간에 새 데이터 추가되면 중복·누락 가능.
  - **cursor** = "이 책갈피부터 20개" (북마크 방식). 더 안정적. 무한 스크롤에 유리.
- 💡 **limit** = 한 번에 받을 개수 (보통 20~50개).

**응답 예시:**
> 💡 **응답** = 서버가 돌려주는 내용.
> 💡 **JSON** = 데이터를 표현하는 형식. 중괄호 `{}`로 감싸고 `"키": 값` 쌍으로 표기. 거의 모든 API가 이 형식 사용.

```json
{
  "items": [
    {
      "id": "p1",
      "name": "Sari Dewi",
      "partnerType": "helper",
      "lat": -6.2440, "lng": 106.8022,
      "photo": "https://cdn.linka.id/u/p1.jpg",
      "location": "Kebayoran Baru",
      "rating": 5.0,
      "totalJobs": 312,
      "pricePerHour": 30000,
      "isAvailable": true,
      "isVerified": true,
      "experienceYears": 10,
      "serviceFrequency": "regular",
      "skills": ["Beberes", "Masak", "Cuci"]
    }
  ],
  "nextCursor": "eyJvZmZzZXQiOjIwfQ==",
  "total": 25
}
```

**응답 해석:**
- `items` = 결과 배열. 각 파트너 한 명씩 들어있음.
- `nextCursor` = 다음 페이지 가져올 때 쓸 책갈피 (다음 호출 시 `cursor=eyJv...` 로 보냄). 더 없으면 null.
- `total` = 전체 결과 수 (페이지 합).
- `"eyJvZmZzZXQiOjIwfQ=="` 같은 알 수 없는 글자 = **Base64 인코딩**된 값. 사람이 읽을 수 없게 변환한 것 (속을 알 필요 없음, 그대로 다시 보내기만 하면 됨).

> ⚠️ **성능 고려**: 마커용 데이터는 가벼워야 함. 매우 많은 마커가 예상되면 **마커용/카드용 엔드포인트 분리** 또는 **필드 셀렉터** 도입 검토.
> 💡 **필드 셀렉터** = "필요한 필드만 골라서 줘" 옵션. 마커는 id·lat·lng·photo만, 카드는 전체 필드.

### 7.2 단일 파트너 상세 🤝

```
GET /api/v1/partners/{id}
```

> 💡 **`{id}`** = 자리 표시자. 실제로는 `/api/v1/partners/p1` 처럼 그 자리에 파트너 ID가 들어감. **Path 파라미터**라고 부름. Query 파라미터(`?id=p1`)와 구분.

- 미리보기·상세 화면에서 공통 사용
- 응답: 위 + 자기소개, 후기 요약, 가능 시간대 등

---

### 7.3 활동 카테고리 마스터 🤝

```
GET /api/v1/activities
```

- 다국어 라벨 (id/ko/en/zh/ja) 포함
- 백엔드와 프론트가 동일한 카테고리·항목·skill 매핑을 공유해야 함

**응답 예시:**
```json
[
  {
    "id": "cleaning",
    "label": { "id": "Kebersihan·Kerapian", "ko": "청소·정리", "en": "Cleaning" },
    "items": [
      { "id": "floor", "label": {...}, "skill": "Beberes" }
    ]
  }
]
```

**응답 해석:**
- 카테고리 `cleaning` 안에 `items` (세부 항목들)
- `label` 안에 언어별 표시명 (`id`=인니어, `ko`=한국어 등)
- 각 항목은 `skill` (백엔드가 헬퍼 검색할 때 쓰는 코드)에 매핑

> ⚠️ **합의 필요**: 카테고리 마스터를 **클라이언트 하드코딩**으로 둘지, **서버 관리**(원격 설정)로 둘지.
> 💡 **하드코딩** = 코드 안에 값을 박아놓기. 카테고리 추가하려면 앱 업데이트 필요.
> 💡 **원격 설정** = 서버에서 가져오기. 카테고리 추가 시 앱 업데이트 없이 바로 반영. 신규 카테고리 자주 추가될 거면 이쪽.

---

### 7.4 사용자 위치 (참고)

- 디바이스 GPS는 `expo-location`으로 클라이언트가 직접 획득. 서버에 보낼 필요는 없으나, "내 주변 검색"은 위 7.1의 `lat/lng/radius`로 처리.

> 💡 **expo-location** = Expo 라이브러리 중 하나. 폰의 GPS 권한 받고 위치 가져옴.

## 8. 비즈니스 로직 / 정책 이슈

> 💡 이 섹션은 백엔드 회의 안건. 각 항목이 "이거 어떻게 할까요?" 결정 포인트.

⚠️ **다음 항목은 백엔드와 합의 필요:**

1. **`isAvailable` 정의**
   - 파트너 토글? (본인이 "지금 가능" 직접 표시)
   - 스케줄 기반 자동? (예약 시간표 보고 시스템이 판단)
   - 둘 다?

2. **위치 갱신 주기**
   - 등록된 활동 거점인지 (한 번 정해놓은 동네)
   - 실시간 GPS인지 (배달앱처럼 움직임)
   - 지금 프론트는 정적 가정 (한 번 정해놓은 위치)

3. **거리 정렬 미지원**
   - 현재 정렬: 평점/후기/가격
   - 거리순 정렬 필요? — 필요하면 7.1 응답에 `distanceMeters` 필드 추가

4. **검증(`isVerified`) 기준**
   - 어떤 서류·절차 통과 시 true?
   - KTP 사진만? 면접까지? 운영자 승인?

5. **헬퍼/드라이버 통합 검색 vs 분리**
   - 현재는 한 엔드포인트에서 `partnerType` 필터로 구분
   - 드라이버 전용 필드(차종·면허·서비스종류)가 많아지면 엔드포인트 분리 검토

6. **활동 → skill 매핑**
   - 프론트가 매핑할지 (현재 방식): 사용자가 "바닥 청소" 선택 → 프론트가 "Beberes"로 변환 → 서버에 Beberes 전달
   - 서버가 매핑할지: 프론트가 "floor" 그대로 보내면 서버가 변환

7. **검색 빈도/캐싱**
   - 필터 변경마다 API 호출하면 비용 큼
   - **디바운싱** + **일정 영역 캐싱** 정책 합의
   - 💡 **디바운싱** = 사용자가 빨리 여러 번 입력해도 한 번만 처리. "0.5초 기다리고 변화 없으면 그때 검색".
   - 💡 **캐싱** = 한 번 받은 결과를 임시 저장해서 다시 안 부르기. 같은 영역 다시 보면 캐시에서 즉시 표시.

8. **다국어 location 표기**
   - `location: "Kebayoran Baru"` 같은 동네명을 한국어로도 보여줄지 (현재는 인니어 그대로)

## 9. 엣지 케이스 / 빈 상태

> 💡 **엣지 케이스** = 정상 케이스가 아닌 특수 상황. "데이터 없으면?", "에러 나면?" 등.
> 💡 **빈 상태 (Empty State)** = 보여줄 데이터가 없는 화면 상태. 사용자가 길을 잃지 않도록 안내 메시지 + 다음 행동 버튼 두는 게 중요.

| 상황 | 화면 동작 |
|---|---|
| 위치 권한 거부 | 기본 데모 영역(Kebayoran Baru) 표시. "내 위치" 버튼 비활성 |
| 파트너 0건 | 빈 상태 메시지 + 필터 초기화 버튼 (현재 미구현) |
| 네트워크 에러 | 재시도 토스트 (현재 미구현) |
| 검증되지 않은 파트너 | 카드/마커에서 검증 배지 미노출 |
| 즉시 가능하지 않은 파트너 | "예약 불가" 표기, 예약 버튼 비활성 |
| 매우 많은 마커 (100+) | 마커 클러스터링 필요 여부 검토 |

> 💡 **토스트** = 화면 아래에 잠시 떠올랐다가 사라지는 메시지 (안드로이드에서 자주 보는 그것).
> 💡 **클러스터링** = 너무 많은 마커가 한 곳에 모여있을 때 "20" 같은 숫자 묶음으로 보여주고, 줌인하면 풀어지는 처리.
