# Linka 프론트엔드 담당자 가이드

> 비개발자가 Linka 앱 프론트엔드를 책임지고, 백엔드 팀과 소통하며 마무리하기 위한 최소 필요 지식.

---

## 1. 내 역할 한 줄 요약

> **Linka 모바일 앱(iOS/Android)의 화면을 만들고, 백엔드와 연결해 출시까지 책임진다.**

- 백엔드: 외주(Java + AWS + PostgreSQL)
- 관리자 웹: 별도 (React + Vite + Tailwind, 담당 미정)
- **모바일 프론트: 본인**

---

## 2. 내가 책임지는 것

| 작업 | 대략적인 시점 |
|---|---|
| 화면 UI 구현 (이미 80% 완료) | 지금 |
| 백엔드 API 스펙 합의 | 백엔드 첫 회의 |
| 인증·토큰 처리 | API 1차 연동 |
| Mock 데이터를 실제 API로 교체 | API 받는 대로 |
| 로딩·에러 화면 처리 | API 연동 시 |
| 빌드·배포 (TestFlight, Play Store) | 출시 직전 |
| 버그 수집·버전 관리 | 출시 후 |

## 3. 내가 알아야 하는 것 (큰 그림)

### A. 자기 코드 읽기
- 폴더 구조, 화면 찾기 → [01-앱-한눈에](./01-앱-한눈에.md)
- TSX 코드 줄 단위 이해 → [02-tsx-읽는법](./02-tsx-읽는법.md)
- 컴포넌트 / Props / State → 03 (작성 예정)

### B. 화면 단위 작업
- 새 화면 추가, 네비게이션 → 04 (작성 예정)
- 스타일링 (StyleSheet) → 05 (작성 예정)
- 다국어 텍스트 추가 → 06 (작성 예정)

### C. 백엔드와 소통
- API 스펙 합의 (PPT 기획안 사용)
- 자주 나오는 용어 (CORS, 401, rate limit, JWT 등) → [용어집](./용어집.md)
- 질문 템플릿 (API 안 됨 / 응답 이상함 / 필드 추가 요청)
- 회의 체크리스트

### D. API 연동
- TanStack Query (서버 데이터 관리 라이브러리) 도입
- axios (HTTP 통신) + 인터셉터로 토큰·언어 헤더 자동 처리
- 로딩 / 에러 / 성공 세 가지 상태 UI

### E. 디버깅 생존 기술
- "왜 안 되지?" 일 때 확인 순서: 네트워크 탭 → 콘솔 → 백엔드 로그
- 자주 나는 에러 5개와 대처
- 백엔드 vs 프론트 책임 구분하는 법

### F. Git 협업
- 커밋·브랜치·푸시·복원 4가지 명령어
- PR 만들고 머지하기
- 충돌 무서워하지 않기

### G. 빌드·배포
- TestFlight (iOS 베타)
- Google Play 내부 테스트
- Expo OTA (코드만 바뀌면 앱 업데이트 없이 배포)

## 7. 막혔을 때

1. **무엇이 안 되는지 한 줄로 적어보기** — "지도에서 마커 클릭하면 상세화면이 안 떠"
2. **콘솔(에러 메시지) 확인** — 빨간 글자가 답인 경우 70%
3. **AI에게 물어보기** — Claude Code, ChatGPT 등에 코드 + 에러 메시지 같이 던지기
4. **백엔드 책임인지 의심해보기** — Network 탭에서 응답 코드가 4xx/5xx면 백엔드 쪽
5. **그래도 안 되면 백엔드 동료에게 정리해서 질문**

## 8. 핵심 메시지

> **모든 걸 알 필요 없어요. 막혔을 때 누구에게 어떻게 물어볼지만 알면 충분합니다.**

비개발자가 프론트를 책임진다는 건 **번역가** 역할에 가까워요:
- 디자이너 ↔ 코드
- 백엔드 ↔ 사용자 화면
- 비즈니스 요구 ↔ 기술 결정

이 번역에 필요한 단어와 패턴만 익혀두면 충분히 굴러갑니다.
